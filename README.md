# بوت مضمون برو للخدمات الرقمية

نواة متجر تيليجرام عربية وآمنة لبيع الاشتراكات والخدمات الرقمية. المشروع يفصل بين واجهة العميل، المحاسبة، الدفع، والمورد؛ لذلك يمكن تغيير المورد أو إضافة وسيلة دفع من دون إعادة كتابة البوت.

> الحالة الحالية: نسخة تأسيسية قابلة للتشغيل والاختبار. كل التكاملات التي تحتاج أموالًا حقيقية تكون معطلة افتراضيًا إلى أن تُضاف حسابات التاجر وتُختبر الاستجابات الفعلية.

## ما الذي يعمل

- متجر بأقسام وخدمات وأسعار قابلة للتعديل.
- محفظة USDT داخلية وسجل قيود غير قابل للتعديل.
- منع تكرار الخصم أو الشحن بمفاتيح idempotency.
- طلبات يدوية أو آلية، وتسليم مشفر داخل قاعدة البيانات.
- شحن يدوي لجيب والكريمي: سعر صرف، رسوم، رقم عملية، صورة إثبات، اعتماد أو رفض من الأدمن.
- Binance Pay Merchant: إنشاء طلب دفع، توقيع HMAC-SHA512، والتحقق من توقيع webhook بواسطة شهادة RSA قبل إضافة الرصيد.
- موصل VenteBot باستخدام `X-Reseller-Key`، وهو معطل حتى مطابقة حقول إنشاء الطلب مع ملف OpenAPI الكامل.
- لوحة أدمن داخل تيليجرام لإدارة الخدمات والأسعار والأقسام والأرصدة وطرق الدفع والطلبات.
- PostgreSQL وRedis وDocker، مع SQLite وذاكرة محلية للتطوير.
- تشفير بيانات العميل وبيانات التسليم بواسطة Fernet.

## قواعد أمان مهمة

1. لا تضع التوكن أو مفاتيح الدفع داخل الكود، ولا ترسلها في المحادثات. ضعها فقط في ملف `.env` على الخادم.
2. استخدم حساب **Binance Pay Merchant**. عنوان USDT أو مفاتيح التداول الشخصية لا تكفي ولا يجوز استخدامها بدل بوابة التاجر.
3. لا تطلب من العميل كلمة مرور Google أو رمز التحقق. لخدمة Gemini اطلب البريد فقط، واستخدم توريدًا مرخصًا ومتوافقًا مع شروط Google والمورد.
4. لا نشغّل حساب تيليجرام آليًا لسحب الطلب من بوت المورد. الربط يكون عبر API رسمي فقط.
5. إذا كانت نتيجة إرسال طلب المورد غير مؤكدة بسبب انقطاع الشبكة، ينتقل الطلب إلى المراجعة ولا يعاد تلقائيًا؛ هذا يمنع شراء نفس الطلب مرتين.
6. لا تفعّل جيب أو الكريمي قبل ضبط رقم الحساب وسعر الصرف. الربط التلقائي لا يُفعل إلا بعد عقد تاجر ووثائق API رسمية.

## تشغيل محلي سريع

المتطلبات: Python 3.12 و`uv`.

```bash
cp .env.example .env
uv sync
uv run python -m app.cli generate-key
```

انسخ المفتاح الناتج إلى `DATA_ENCRYPTION_KEY` في `.env`، ثم أضف توكن BotFather ورقم Telegram ID للأدمن. للتجربة السريعة غيّر قاعدة البيانات في `.env` إلى:

```env
DATABASE_URL=sqlite+aiosqlite:///./store.db
APP_ENV=development
```

شغّل التطبيق:

```bash
uv run uvicorn app.main:app --reload
```

افتح البوت وأرسل `/start`، ثم `/admin` من حساب الأدمن.

## التشغيل بواسطة Docker

1. انسخ `.env.example` إلى `.env` واضبط القيم.
2. اجعل `POSTGRES_PASSWORD` وكلمة المرور داخل `DATABASE_URL` متطابقتين.
3. شغّل:

```bash
docker compose up -d --build
docker compose logs -f app
```

في الإنتاج يستخدم البوت Telegram Webhook على
`/telegram/<WEBHOOK_SECRET_PATH>`، ولذلك تستيقظ خدمة الاستضافة عند وصول رسالة.
في التطوير المحلي فقط يستخدم البوت long polling.

يوجّه Docker المنفذ إلى `127.0.0.1:8000`. ضع أمامه Caddy أو Nginx مع HTTPS، واضبط `PUBLIC_BASE_URL` على النطاق العام.

## إعداد المتجر من لوحة الأدمن

ترتيب الإعداد المقترح:

1. أضف الأقسام والخدمات أو عدّل خدمة Gemini Pro النموذجية.
2. اضبط السعر ومعرّف منتج المورد ونوع التنفيذ، ثم فعّل الخدمة.
3. افتح «طرق الدفع»، واضبط سعر صرف الكريمي/جيب وتعليمات التحويل، ثم فعّل الطريقة.
4. نفّذ شحنًا تجريبيًا بقيمة صغيرة وتأكد أن الاعتماد الثاني لا يضيف الرصيد مرة أخرى.
5. نفّذ طلبًا يدويًا، ثم جرّب التسليم ورد الرصيد.
6. بعد نجاح اختبارات المورد وBinance في بيئة تجريبية فقط، غيّر مفاتيح التفعيل في `.env`.

## تفعيل المورد

الصور تثبت:

- العنوان الأساسي: `https://ventetelegrambotrailway-production.up.railway.app`
- المصادقة: `X-Reseller-Key` (ويدعم `X-API-Key` للتوافق)
- مسارات الحساب والمنتجات والتسعير والطلبات وحركات المحفظة.
- حد الطلبات: 60 طلبًا خلال 60 ثانية لكل مفتاح.

لكن مخططات الحقول في الصور مطوية. قبل جعل `SUPPLIER_ENABLED=true` نحتاج ملف
`/api/reseller/openapi.json` أو صور المخططات مفتوحة، خصوصًا:

- المسار والطريقة HTTP.
- حقول `product_id` وبيانات العميل والكمية.
- حالات الطلب ورسالة التسليم.
- هل يقبل `client_order_id` لمنع التكرار؟
- كيف نستعلم عن طلب بعد timeout؟
- أخطاء الرصيد والمنتج غير المتاح.

بعد المطابقة اضبط القيم في `.env` واختبر منتجًا رخيصًا بحساب تجريبي.

## تفعيل Binance Pay

اضبط:

```env
BINANCE_PAY_ENABLED=true
BINANCE_PAY_API_KEY=هوية_API_للتاجر
BINANCE_PAY_SECRET_KEY=سر_API_للتاجر
PUBLIC_BASE_URL=https://your-domain.example
WEBHOOK_SECRET_PATH=نص_عشوائي_طويل
```

لا يضيف webhook أي رصيد قبل فحص السر في الرابط، توقيع RSA، رقم طلبنا، العملة، والمبلغ. التكرار آمن لأن قيد الشحن له مفتاح فريد.

## بنية المشروع

```text
app/
  handlers/          واجهات العميل والأدمن
  services/          المحفظة والطلبات والدفع
    payments/        Binance Pay والمحاسبة
    providers/       موصلات الموردين
  web/               health وwebhooks
  models.py          نموذج قاعدة البيانات
  main.py            تشغيل HTTP والبوت والعامل الخلفي
tests/               اختبارات المحاسبة والتواقيع والطلبات
docs/                التصميم وقائمة التفعيل
```
