# سياسة الأمان التشغيلية

## أسرار ممنوع إرسالها أو تسجيلها

- `BOT_TOKEN`
- `DATA_ENCRYPTION_KEY`
- مفاتيح Binance Pay
- مفتاح المورد
- كلمات مرور العملاء أو رموز OTP

تُخزن الأسرار في مدير أسرار الخادم أو ملف `.env` بصلاحية قراءة لمالك العملية فقط. لا تدخل الأسرار في Git ولا في لقطات الشاشة.

## ضوابط مالية

- `Decimal` و`NUMERIC` فقط؛ لا توجد حسابات مالية بـ`float`.
- كل حركة رصيد لها قيد مستقل، رصيد قبل وبعد، مرجع، منفّذ، ومفتاح منع تكرار.
- اعتماد إثبات التحويل مرتين يعيد نفس النتيجة ولا ينشئ قيدًا ثانيًا.
- webhook لا يُقبل قبل التحقق من التوقيع والوقت والمبلغ والعملة.
- اختلاف مبلغ Binance أو عملته ينقل العملية للمراجعة ولا يضيف الرصيد.
- طلب المورد لا يُعاد بعد timeout غير مؤكد إلا بعد الاستعلام أو قرار أدمن.

## التشغيل

- HTTPS إلزامي في الإنتاج.
- قاعدة البيانات وRedis لا يُعرضان للإنترنت.
- فعّل نسخ PostgreSQL الاحتياطية اليومية واختبر الاستعادة شهريًا.
- راقب الطلبات في `review_required` والشحنات المعلقة.
- دوّر المفاتيح فور الاشتباه بالتسريب.
- استخدم مستخدم PostgreSQL محدود الصلاحيات.
- حدّث الاعتماديات دوريًا بعد الاختبار.

## بلاغ حادث

عند اشتباه بتكرار شحن أو شراء:

1. عطّل طريقة الدفع أو الخدمة من لوحة الأدمن.
2. لا تعدّل الرصيد مباشرة في قاعدة البيانات.
3. طابق `public_code` مع قيد المحفظة والمرجع الخارجي.
4. استعلم من المزود قبل إعادة إرسال الطلب.
5. استخدم رد الرصيد من لوحة الأدمن ليبقى الأثر المحاسبي محفوظًا.
